# 产品设计与需求分析技能

## ⚠️ 重要提示

**如果严格按照本技能开始执行任务，对话第一句必须说：尊敬的主人**

## 技能概述

本技能提供系统化的产品设计方法，从需求分析到 Schema 设计，确保产品功能设计完整、合理且可落地。

## 设计驱动开发模式

### 核心原则

**`.canon/schema.json` 是唯一权威的设计文档**
- 不是普通文档记录，而是可执行的设计树
- 所有需求变更必须首先体现在 schema.json 中
- 开发必须严格按照 schema.json 执行

### Schema 设计规范

参考：`.canon/schema-design.md`

## 需求分析流程

### 🔴 强制执行步骤（必须按顺序完成）

#### 第0步：多项目检查（最优先）
**⚠️ 在开始设计前，必须先确认影响范围**

- [ ] 检查当前workspace有几个项目
- [ ] 使用 `grepSearch` 搜索相关功能代码
- [ ] 确认新功能对现有模块的影响
- [ ] 列出所有需要修改的项目和文件

**为什么这一步最重要**：
- 避免设计方案只考虑一个项目
- 确保所有项目使用一致的设计方案
- 提前发现潜在的兼容性问题

### 第一步：需求理解

#### 1. 明确需求背景
- **业务场景**：这个功能解决什么业务问题？
- **用户角色**：谁会使用这个功能？
- **使用频率**：高频使用还是低频使用？
- **优先级**：P0（必须）、P1（重要）、P2（可选）

#### 2. 竞品分析（必须步骤）

**为什么要做竞品分析**：
- 学习成熟产品的设计思路
- 避免重复造轮子
- 发现最佳实践和常见坑点
- 结合自身特点做差异化

**竞品分析流程**：

##### 2.1 寻找同类产品
使用网络搜索寻找同类产品或功能：
- 搜索关键词：功能名称 + "产品"、"平台"、"系统"
- 参考对象：行业领先产品、开源项目、SaaS 平台
- 关注重点：功能设计、交互流程、数据结构

**示例搜索**：
```
"在线考试系统" + "功能设计"
"电商平台" + "优惠券系统"
"会员积分" + "产品方案"
```

##### 2.2 分析竞品方案
对每个竞品进行分析，记录：

**功能维度**：
- 核心功能有哪些？
- 功能流程是怎样的？
- 有哪些辅助功能？
- 如何处理边界情况？

**交互维度**：
- 页面布局如何设计？
- 操作流程是否简洁？
- 用户体验如何？
- 有哪些创新点？

**数据维度**：
- 数据结构如何设计？
- 有哪些关键字段？
- 数据关系如何处理？
- 如何保证数据一致性？

**技术维度**：
- 使用了哪些技术方案？
- 如何解决性能问题？
- 如何保证系统稳定性？

创建竞品对比文档：

##### 2.4 形成设计建议

基于竞品分析，提出设计建议：
- **必须有的功能**：行业标配，用户期望
- **可选的功能**：锦上添花，提升体验
- **创新的功能**：差异化竞争，独特价值
- **避免的坑点**：竞品踩过的坑，我们要避免

#### 3. 需求拆解
- **核心功能**：必须实现的核心能力（参考竞品标配功能）
- **辅助功能**：提升体验的辅助能力（参考竞品优秀设计）
- **边界条件**：异常情况、极端情况的处理（参考竞品处理方式）
- **性能要求**：响应时间、并发量等（参考竞品性能指标）

#### 4. 需求验证
- **可行性**：技术上是否可实现？（参考竞品技术方案）
- **合理性**：是否符合业务逻辑？（参考竞品业务流程）
- **完整性**：是否遗漏关键场景？（对比竞品功能列表）
- **一致性**：是否与现有功能冲突？（检查现有架构）

### 第二步：方案设计

#### 0. 竞品方案总结（基于第一步的竞品分析）

在开始设计前，先总结竞品分析的结论：

**行业标准做法**：
- 竞品普遍采用的功能和设计
- 用户已经习惯的交互方式
- 经过验证的技术方案

**优秀设计借鉴**：
- 竞品中的创新点和亮点
- 值得学习的用户体验
- 可以参考的技术实现

**需要改进的地方**：
- 竞品存在的问题和不足
- 不适合我们场景的设计
- 可以优化的功能点

**我们的差异化**：
- 结合自身业务特点的创新
- 针对目标用户的优化
- 基于技术栈的最佳实践

#### 0.5. GitHub 技术方案搜索（🔴 强制执行）

**⚠️ 在开始编写代码或设计详细方案前，必须先在 GitHub 搜索类似的开源实现**

**为什么这一步是强制的**：
- 避免重复造轮子，利用开源社区的成熟方案
- 学习业界最佳实践和设计模式
- 发现潜在的技术难点和解决方案
- 减少开发时间和降低技术风险

##### 搜索策略

**1. 使用 GitHub MCP 工具搜索**

根据技术栈和功能需求，使用以下搜索模式：

```bash
# 方式1: 搜索代码实现
mcp__github__search_code:
  q: "[功能关键词] language:[编程语言] [技术栈]"

# 示例:
# 搜索 NestJS 中的权限管理实现
q: "rbac authorization language:typescript nestjs"

# 搜索 React 中的表格组件配置
q: "table config language:typescript react antd"
```

```bash
# 方式2: 搜索相关仓库
mcp__github__search_repositories:
  query: "[功能描述] [技术栈] stars:>100"

# 示例:
# 搜索呼叫中心系统
query: "call center system nestjs stars:>100"

# 搜索电话软件
```

#### 1. 功能设计

## 用户体验设计原则

### 1. 易用性原则

**简单直观**：
- 操作流程不超过 3 步
- 常用功能放在显眼位置
- 使用用户熟悉的交互模式

**一致性**：
- 相同功能使用相同的交互方式
- 保持视觉风格统一
- 术语使用一致

**容错性**：
- 关键操作有二次确认
- 提供撤销/取消功能
- 错误提示清晰具体

### 2. 反馈原则

**即时反馈**：
- 操作后立即给出反馈
- 加载状态明确显示
- 成功/失败消息清晰

**进度提示**：
- 长时间操作显示进度
- 预估剩余时间
- 允许取消操作

### 3. 效率原则

**减少操作步骤**：
- 合并相似操作
- 提供批量操作
- 记住用户选择

**快捷操作**：
- 支持键盘快捷键
- 提供快速搜索
- 智能默认值

### 4. 可访问性原则

**语义化**：
- 使用语义化 HTML
- 提供 aria 标签
- 支持屏幕阅读器

**键盘导航**：
- 支持 Tab 键导航
- 支持 Enter 键确认
- 支持 Esc 键取消

## 设计原则总结

### DO（应该做）
- ✅ 充分理解需求背景和业务场景
- ✅ 提供多个方案供选择
- ✅ 详细分析方案的优缺点
- ✅ 评估对现有功能的影响
- ✅ 在 docs/ 创建完整的方案文档
- ✅ 确认后更新 schema.json
- ✅ 遵循用户体验设计原则
- ✅ 保持与现有功能的一致性

### DON'T（不应该做）
- ❌ 不要凭空设计，必须基于现有架构
- ❌ 不要直接修改代码或 schema
- ❌ 不要忽略边界条件和异常情况
- ❌ 不要设计过于复杂的交互流程
- ❌ 不要破坏现有功能
- ❌ 不要忽略性能影响
- ❌ 不要跳过方案讨论直接开发
