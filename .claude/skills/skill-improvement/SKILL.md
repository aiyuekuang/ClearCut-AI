# 技能更新与改进技能

## ⚠️ 重要提示

**如果使用了本技能，对话第一句必须说：尊敬的主人**

## 技能概述

本技能用于持续改进和更新技能文档，将开发过程中的经验、教训和最佳实践沉淀到技能体系中，确保团队知识不断积累和传承。

## 核心原则

### 1. 经验驱动
- 从实际问题中总结经验
- 记录真实的错误案例
- 提炼可复用的解决方案

### 2. 持续改进
- 定期回顾和更新技能文档
- 根据实践反馈优化流程
- 删除过时或无效的内容

### 3. 知识共享
- 将个人经验转化为团队知识
- 用清晰的语言描述问题和解决方案
- 提供可操作的步骤和示例

## 技能更新流程

### 🔴 强制执行步骤（必须按顺序完成）

#### 第0步：识别改进点
**⚠️ 在开始更新前，必须先明确改进目标**

- [ ] 确定要改进的技能类型（Bug修复/产品设计/代码质量等）
- [ ] 明确改进的原因（遇到的问题、缺失的内容、需要优化的流程）
- [ ] 收集相关的案例和经验
- [ ] 评估改进的影响范围

**触发场景**：
- 遇到技能文档未覆盖的问题
- 发现技能流程存在缺陷
- 积累了新的最佳实践
- 用户反馈技能文档不够清晰

#### 第1步：分析现有技能
**深入理解当前技能的内容和结构**

1. **读取目标技能文档**
   ```bash
   # 读取技能文档
   readFile: path=".claude/skills/[技能名称]/SKILL.md"
   ```

2. **分析技能结构**
   - 技能概述是否清晰
   - 流程步骤是否完整
   - 检查清单是否全面
   - 示例是否充分
   - 常见错误是否覆盖

3. **识别缺失内容**
   - 哪些场景没有覆盖
   - 哪些步骤不够详细
   - 哪些错误模式未记录
   - 哪些最佳实践未提及

#### 第2步：设计改进方案
**制定具体的改进内容**

1. **新增内容**
   - 新的流程步骤
   - 新的检查项
   - 新的错误模式
   - 新的最佳实践
   - 新的示例代码

2. **优化内容**
   - 简化复杂的描述
   - 补充不清晰的说明
   - 更新过时的信息
   - 调整不合理的流程

3. **删除内容**
   - 过时的技术方案
   - 不再适用的规范
   - 重复的内容

#### 第3步：编写改进内容
**按照技能文档规范编写**

**内容要求**：
- ✅ 使用清晰的标题和层级
- ✅ 提供具体的步骤和示例
- ✅ 标注重要提示（⚠️、✅、❌）
- ✅ 包含检查清单
- ✅ 添加常见错误模式（如适用）

**示例格式**：

```markdown
### 新增步骤：[步骤名称]

**为什么需要这一步**：
- 原因1
- 原因2

**执行步骤**：
1. 步骤1
2. 步骤2

**检查清单**：
- [ ] 检查项1
- [ ] 检查项2

**常见错误**：
- ❌ 错误做法
- ✅ 正确做法
```

#### 第4步：更新技能文档
**将改进内容写入技能文档**

1. **使用 strReplace 更新**
   - 精确定位要修改的位置
   - 保持原有格式和风格
   - 确保修改后的内容连贯

2. **更新版本号**
   - 小改进：1.0.0 → 1.0.1
   - 新增功能：1.0.0 → 1.1.0
   - 重大重构：1.0.0 → 2.0.0

3. **添加更新记录**
   - 在技能文档末尾添加更新日志
   - 记录更新时间、内容和原因

**⚠️ 更新后必须观察结果（强制执行）**：
```bash
# 1. 验证文件已更新
readFile: path=".claude/skills/[技能名称]/SKILL.md"
# 检查更新的内容是否正确

# 2. 检查是否有语法错误
# - Markdown 格式是否正确
# - 代码块是否闭合
# - 链接是否有效

# 3. 如果发现问题，立即修复
# 不要留下错误的内容
```

#### 第5步：同步到所有项目
**确保所有workspace使用最新技能**

- [ ] 检查有几个项目有 `.claude/skills` 目录
- [ ] 将更新后的技能文档同步到所有项目
- [ ] 如果涉及 all.md，同步更新所有项目的 all.md

**⚠️ 同步后必须观察结果（强制执行）**：
```bash
# 1. 验证所有项目的文件已更新
readFile: path="[项目名]/.claude/skills/[技能名称]/SKILL.md"
# 逐个检查每个项目

# 2. 确认内容一致
# 使用 diff 或直接对比内容

# 3. 如果有项目未同步，立即补充
```

#### 第6步：验证改进效果
**确认改进达到预期目标**

- [ ] 重新阅读更新后的技能文档
- [ ] 检查是否解决了原始问题
- [ ] 确认内容清晰易懂
- [ ] 验证示例代码正确
- [ ] 测试流程是否可执行

## 常见改进场景

### 场景1：添加新的错误模式

**触发条件**：遇到了技能文档未记录的典型错误

**改进步骤**：
1. 在技能文档中找到"常见错误模式"章节
2. 添加新的错误模式，包含：
   - 错误症状
   - 错误原因
   - 错误代码示例
   - 正确代码示例
   - 预防措施

**示例**：
```markdown
### 错误N：[错误名称]
**症状**：[问题表现]

**错误代码**：
\`\`\`typescript
// ❌ 错误示例
\`\`\`

**正确代码**：
\`\`\`typescript
// ✅ 正确示例
\`\`\`

**预防措施**：
- 措施1
- 措施2
```

### 场景2：优化流程步骤

**触发条件**：发现现有流程存在缺陷或可以优化

**改进步骤**：
1. 找到需要优化的流程步骤
2. 分析问题所在
3. 设计改进方案
4. 更新流程描述
5. 更新检查清单

### 场景3：补充最佳实践

**触发条件**：积累了新的开发经验和最佳实践

**改进步骤**：
1. 在技能文档中找到合适的位置
2. 添加最佳实践说明
3. 提供示例代码
4. 说明为什么这样做更好

### 场景4：更新技术方案

**触发条件**：技术栈升级或采用新的技术方案

**改进步骤**：
1. 标记过时的内容
2. 添加新的技术方案
3. 说明新旧方案的差异
4. 提供迁移指南（如需要）

### 场景5：简化复杂描述

**触发条件**：发现某些内容过于复杂或难以理解

**改进步骤**：
1. 识别复杂的描述
2. 用更简单的语言重写
3. 添加图表或示例
4. 分解为更小的步骤

## 技能文档规范

### 1. 文件结构

```markdown
---
name: 技能名称
description: 技能描述
version: 版本号
author: 作者
tags:
  - 标签1
  - 标签2
---

# 技能名称

## ⚠️ 重要提示
**问候语要求**

## 技能概述
[技能简介]

## 核心原则/流程
[主要内容]

## 检查清单
[检查项]

## 常见错误模式（如适用）
[错误案例]

## 相关资源
[参考链接]

## 更新日志
[更新记录]
```

### 2. 格式规范

**标题层级**：
- `#` 一级标题：技能名称
- `##` 二级标题：主要章节
- `###` 三级标题：子章节
- `####` 四级标题：详细说明

**强调标记**：
- `⚠️` 重要警告
- `✅` 正确做法
- `❌` 错误做法
- `🔴` 强制要求
- `[ ]` 检查项

**代码块**：
- 使用三个反引号包裹
- 标注语言类型
- 添加注释说明

**列表**：
- 有序列表：使用数字
- 无序列表：使用 `-`
- 检查清单：使用 `- [ ]`

### 3. 内容原则

**清晰性**：
- 使用简单直白的语言
- 避免模糊的描述
- 提供具体的步骤

**完整性**：
- 覆盖常见场景
- 包含正反示例
- 提供检查清单

**可操作性**：
- 步骤具体可执行
- 示例代码可运行
- 检查项可验证

**实用性**：
- 基于真实案例
- 解决实际问题
- 提供最佳实践

## 通用规范：终端操作观察

**⚠️ 所有技能文档中涉及终端操作的地方，都必须遵守以下规范**

### 核心原则
**执行任何终端命令后，必须立即观察输出并处理错误**

### 观察流程（强制执行）

#### 1. 执行命令
```bash
# 执行任何终端命令
executeBash: command="..."
# 或
controlBashProcess: action="start", command="..."
```

#### 2. 观察输出（必须）
```bash
# 立即检查命令输出
# - 查看返回码（0表示成功）
# - 查看输出内容
# - 查找错误信息
# - 查找警告信息
```

#### 3. 分析结果（必须）
```bash
# 判断命令是否成功
# - 成功：继续下一步
# - 失败：分析错误原因
# - 警告：评估是否需要处理
```

#### 4. 处理错误（如有）
```bash
# 如果发现错误，必须：
# 1. 停止当前流程
# 2. 分析错误原因
# 3. 修复错误
# 4. 重新执行命令
# 5. 再次观察输出
# 6. 确认成功后继续
```

#### 5. 持续观察（长期运行的进程）
```bash
# 对于启动服务等长期运行的命令：
# 1. 等待5秒后首次观察
# 2. 检查启动过程中的错误
# 3. 等待10秒后再次观察
# 4. 确认服务稳定运行
# 5. 如有问题，立即停止并修复
```

#### 6. 清理终端（强制执行）
```bash
# 对于一次性执行的命令（非长期运行）：
# 执行完成并观察结果后，必须关闭终端保持整洁

# 示例：执行 git pull
executeBash: command="git pull"
# 观察输出，确认成功
# 如果是一次性命令，无需保持终端打开

# 对于长期运行的进程（服务启动）：
# 使用 controlBashProcess 启动，保持进程运行
# 不要使用 executeBash 启动长期运行的服务
```

**终端管理原则**：
- ✅ 一次性命令（git、npm install、docker build）：执行完成后自动关闭
- ✅ 长期运行的服务（npm run dev、docker compose up）：使用 controlBashProcess 管理
- ❌ 不要让已完成的一次性命令占用终端
- ❌ 不要使用 executeBash 启动长期运行的服务

### 常见终端操作的观察要点

#### Docker 构建
```bash
docker build -t image-name .
# 观察要点：
# - 是否显示 "Successfully built"
# - 是否有编译错误
# - 是否有依赖下载失败
# - 验证镜像已创建：docker images
```

#### Docker 启动
```bash
docker compose up -d
# 观察要点：
# - 等待5秒后检查容器状态：docker compose ps
# - 查看日志：docker compose logs --tail=50
# - 检查端口监听：netstat -tlnp
# - 测试服务响应：curl
# - 持续观察10秒，确认稳定
```

#### Git 操作
```bash
git pull
# 观察要点：
# - 是否显示 "Already up to date"
# - 是否有冲突提示
# - 是否有权限错误
# - 如有冲突，必须解决后继续
```

#### npm/yarn 安装
```bash
npm install
# 观察要点：
# - 是否有依赖冲突
# - 是否有编译错误
# - 是否有权限问题
# - 是否有网络超时
```

#### 进程启动
```bash
npm run dev
# 观察要点：
# - 等待5秒后检查输出
# - 是否有编译错误
# - 是否有端口占用
# - 是否显示启动成功
# - 持续观察10秒，确认稳定
```

### 禁止行为
- ❌ 执行命令后不观察输出
- ❌ 发现错误后不修复就继续
- ❌ 只观察一次就结束（长期运行的进程）
- ❌ 假设命令成功而不验证
- ❌ 忽略警告信息
- ❌ 让已完成的一次性命令占用终端
- ❌ 使用 executeBash 启动长期运行的服务
